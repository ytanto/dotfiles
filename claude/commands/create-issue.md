以下の手順でGitHub Issueを対話形式で作成してください。

**引数**: $ARGUMENTS

## 1. 対象リポジトリの判定

`git rev-parse --is-inside-work-tree` でGitリポジトリ内かどうかを確認する。

- **Gitリポジトリ内の場合**: `gh repo view --json nameWithOwner -q .nameWithOwner` で対象リポジトリを特定
- **Gitリポジトリ外の場合**（`.code-workspace`があるフォルダなど）: ユーザーに対象リポジトリを確認する。複数リポジトリに関わる場合は、メインのリポジトリにIssueを作成し、本文で他リポジトリへの影響を記載する

以降、特定したリポジトリを `<repo>` として使用する（`gh`コマンドでは `-R <repo>` オプションを付与）。

## 2. Issueテンプレートの確認

対象リポジトリの `.github/ISSUE_TEMPLATE/` ディレクトリを確認する。

- **テンプレートが複数ある場合**: 一覧を提示し、ユーザーに使用するテンプレートを選択してもらう
- **テンプレートが1つの場合**: そのテンプレートを使用する
- **テンプレートが見つからない場合**: ユーザーにテンプレートの場所や本文の形式を確認する。テンプレートなしでは進めないこと

選択されたテンプレートの内容を読み取り、本文のベースにする。以降のすべてのステップでテンプレートの構成を厳守すること。

## 3. 要件の整理

引数（`$ARGUMENTS`）から要件を把握する。不明点や曖昧な点があればユーザーに質問して明確にする。

また、ユーザー要望へのリンク（Salesforceなど）があるか確認する。リンクがある場合は「Issue の目的・背景」セクションの先頭に `[ユーザー要望](URL)` の形式で記載する。

## 4. Issueの下書き作成

テンプレートをベースに、タイトルと本文を作成する。**シンプルにわかりやすく**書くこと。冗長な説明・補足は入れない。

- タイトルは簡潔かつ具体的に。テンプレートでタイトルの形式が指定されていない場合は、`feat:` `fix:` `chore:` `refactor:` `docs:` などのprefixを付ける
- 本文には**要件レベル(what/why)のみ**を記載する。ユーザーから明示的に指示がない限り、**実装方針・分析・自分発案の提案は書かない**
- テンプレートのセクション構成は厳守する。テンプレートにあるセクション（目的・背景 / 内容 / 完了の定義 / 確認チェックリスト / 対応方針 / テスト設計 / 疑問点・課題 等）は、ユーザーが「不要」と明示しない限り**勝手に削除しない**。記載者向けセクションだけ埋めて、対応者向けセクションはテンプレ初期値のまま残す
- 言葉選びは自然に。「全面リライト」より「デザイン適用」、「PATCH API追加検討」より「保存単位の見直し」など、要件側の表現を使う
- UIに関する記載はFigma/画面の見た目順に合わせる（タブの並び・項目の順序など）

### 書いてよいもの（what/why）

- 目的・背景（要望リンクがあれば先頭に記載）
- 変更内容（外部から見える挙動・仕様レベル）
- 完了の定義（ユーザーから見た挙動・受け入れ条件）
- デザインリンク（Figma URL）

### 書かないもの（実装詳細・自分発案の分析）

以下はユーザーから明示指示がない限り含めないこと：

**実装詳細(how)**
- 修正対象のクラス/ファイル/メソッド名・関連ファイル一覧
- 既存コードへのフック箇所・拡張方針
- 特定の環境変数名・APIエンドポイント名・設定ファイルパス
- 利用するライブラリ・内部モジュール・内部関数名
- Figmaのnode ID / section ID / variant名などの参照ID（FigmaのトップURLだけで十分）
- 「実装方針」「参考実装」「実装手順」などのセクション

**自分発案の分析・提案**
- API設計の検討観点・トレードオフ分析・推奨案
- PR分割案・実装順序の提案
- 「補足」セクションへの自分発案メモ（PR分割しない/しない等の運用方針メモは、ユーザーが書けと言わない限り入れない）

運用上の外部仕様（例: 「環境変数で機能ON/OFF可能」）のように要件の一部として必要な場合は抽象化して書くこと（具体的な変数名は書かない）。

### Issue作成中に分析が出てきたら

ユーザーとの壁打ちの中で実装上の懸念点（API設計の分岐、影響範囲の調査結果など）が出てきても、**Issue本文には書かない**。それらは担当者が「対応方針」「疑問点・課題」セクションに自分で記載するもの。

## 5. ラベルの確認

`gh label list -R <repo>` で既存ラベルを取得し、適切なラベルを提案する。

## 6. ユーザー確認（必須）

**Issue作成前に必ずユーザーの最終確認を取ること。承認されるまで作成しない。**

テンプレートの形式に厳密に沿った状態で、実際に作成されるIssueの内容をそのまま提示する。
提示する際は以下の形式で、Issueに投稿される本文をそのまま見せること：

```
タイトル: <Issueタイトル>
ラベル: <ラベル>
アサイン: @me

---（以下、Issue本文）---

（テンプレートに沿った本文をここにそのまま記載。テンプレートのセクション構成を崩さないこと）
```

修正の要望があれば反映し、再度同じ形式で確認する。

## 7. Issue作成

ユーザーの承認後、以下のコマンドでIssueを作成する：

```bash
gh issue create -R <repo> --title "<タイトル>" --body "<本文>" --label "<ラベル>" --assignee @me
```

作成後、IssueのURLを表示する。
